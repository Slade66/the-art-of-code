一个典型的 Java 对象在内存中通常由三部分组成：对象头、实例数据、对齐填充。

### 对象头（Header）

存储对象的元数据信息，是每个 Java 对象都必须有的固定开销部分。

对象头最小 12（开启指针压缩）或 16 字节，如果没有实例数据，那么一个对象就是 16 字节。

#### Mark Word（标记字段）

在 64 位 JVM 中，对象头中的 Mark Word 通常占用 8 字节，是一个动态的信息存储区域，其内容会根据对象的运行时状态（如锁状态、GC 标记、是否已计算 identityHashCode）而变化。

其中一个关键点是 identityHashCode。它并不会在对象创建时立即生成，而是采用惰性计算策略：只有在首次显式调用 `System.identityHashCode()` 或未被重写的 `Object.hashCode()` 方法时，JVM 才会计算 identityHashCode 并将其写入 Mark Word。

这个惰性计算机制与 Java 的锁优化，特别是偏向锁（Biased Locking），存在空间上的冲突。为了记录持有锁的线程，偏向锁需要在对象的 Mark Word 中存储线程 ID，而这个位置可能与 identityHashCode 的存储位置发生重叠。
- 如果一个对象的 identityHashCode 已经被计算并写入 Mark Word，它将无法进入偏向锁状态。
- 反之，如果对象已经处于偏向锁状态，此时调用 `hashCode()` 方法尝试写入 identityHashCode，由于空间冲突，偏向锁会被撤销，锁通常会直接升级为轻量级锁或重量级锁。

一旦锁升级为轻量级锁或重量级锁，Mark Word 的用途也会随之改变：此时不再用于存储 `hashCode` 或偏向锁的线程 ID，而是用来存储指向线程栈中锁记录（Lock Record）的指针（轻量级锁），或指向监视器对象（Monitor）的指针（重量级锁）。这种设计避免了与 identityHashCode 的存储冲突。

Mark Word 在不同状态下的存储内容如下：
- 无锁状态：存储对象的 identityHashCode（采用惰性计算策略）、GC 分代年龄（4 位，最大值为 15）和锁标志位（表示无锁）。
- 偏向锁状态：存储持有锁的线程 ID、Epoch（纪元值）、GC 年龄和锁标志位（表示偏向锁）。前提是对象尚未计算并写入 identityHashCode，否则无法进入偏向锁状态。
- 轻量级锁状态：存储指向持锁线程栈中锁记录（Lock Record）的指针，以及锁标志位（表示轻量级锁）。
- 重量级锁状态：存储指向 Monitor（监视器）对象的指针和锁标志位（表示重量级锁）。
- GC 标记状态：在垃圾回收期间，用于临时存储 GC 相关信息，如存活标记或转发指针（具体取决于 JVM 实现）。

#### Klass Pointer（类型指针）

开启压缩指针时是 4 字节，没开就是 8 字节。

指向方法区中类元数据（Class 模板）的指针，用于标识该对象的实际类型。JVM 通过这个指针可以定位到对象所属的类、继承结构、方法表等信息。

#### 数组长度（仅数组对象拥有）

数组对象在对象头中会额外多出一个字段（通常 4 字节），用于记录数组的长度。非数组对象则没有这个字段。

### 实例数据（Instance Data）

这部分用于存储对象的所有实例字段（即非 `static` 修饰的成员变量），包括当前类定义的字段以及从所有父类继承而来的字段。通常，父类的字段在内存布局中会排列在子类字段之前。

实例数据的大小取决于对象本身及其所有父类中实例字段的数量与类型。

### 对齐填充（Padding）

现代计算机处理器为了高效访问内存，通常要求数据从特定的地址（如 8 字节的倍数）开始读取。为了满足这一要求，HotSpot JVM 会确保每个对象的总大小（包括对象头和实例数据）是 8 字节的整数倍。如果对象头和实例数据的总和不是 8 的倍数，JVM 会在实例数据末尾填充无意义的字节，使总大小对齐到最近的 8 字节边界。虽然这些填充字节不存储有效信息，但它们有助于提高 CPU 访问内存的效率，避免了因数据未对齐而导致的额外内存访问和性能开销。
