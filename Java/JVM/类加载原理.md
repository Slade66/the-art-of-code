Java 的类加载是指将 `.class` 文件中的二进制数据读入到内存中，并转换成可以被 Java 虚拟机（JVM）使用的 `java.lang.Class` 对象的过程。

类加载的过程可以分为以下几个阶段：

### 加载（Loading）

类加载系统是 JVM 的核心子系统之一，其主要职责是查找并加载类的字节码到 JVM 中。具体而言，当 Java 程序需要使用某个类时，JVM 的类加载器（ClassLoader）会根据类名定位到相应的 `.class` 文件，并将其字节码读入内存。随后，类加载器会根据加载的字节码在方法区（元空间）中构建出该类对应的运行时数据结构，创建一个代表该类的 Class 对象，该对象存储着类的元信息。

**类加载器**

Java 采用不同的类加载器来加载来自不同来源的类。例如，启动类加载器（Bootstrap ClassLoader）负责加载核心类库，扩展类加载器（Extension ClassLoader）负责加载扩展类库，而应用程序类加载器（Application ClassLoader）则负责加载应用程序自身的类。

- 引导（启动）类加载器（Bootstrap ClassLoader）：作为最顶层的类加载器，它由 C++ 编写，是 JVM 自身的一部分，负责加载 JVM 启动时所需的关键核心类库，这些类库通常位于 `<JAVA_HOME>/jre/lib` 目录下。引导类加载器没有父加载器。获取不到它的引用。

- 扩展类加载器 (Extension ClassLoader)：它负责加载 JVM 扩展目录中的类库，该目录通常是 `<JAVA_HOME>/jre/lib/ext`。扩展类加载器的父加载器是引导类加载器。

- 应用程序类加载器 (Application ClassLoader) / 系统类加载器 (System ClassLoader)：该加载器负责加载用户自定义的类以及应用程序 `classpath` 下的类库。它的父加载器是扩展类加载器。

除了以上三种主要的类加载器，开发者还可以通过继承 `java.lang.ClassLoader` 类来实现自定义的类加载器，以满足一些特殊的加载需求，例如从特定的来源加载类、对加载的类进行加密解密等。

**双亲委派机制**

当一个类加载器收到加载类的请求时，它不会立即自行加载，而是将该请求委派给其父加载器处理。这个委派过程会层层向上传递，直至顶层的启动类加载器。只有当父加载器在其加载路径下无法找到所需的类时，子加载器才会尝试自己进行加载。

好处：
- 安全性：有效防止恶意代码替换 JVM 核心类库的可能性。例如，即使用户自己编写一个同名同包的类（如 `java.lang.Object`）来替换或篡改核心库的功能，该请求也会被委派给启动类加载器处理，而启动类加载器只会从其受信任的路径加载，从而确保 JVM 核心类的安全。
- 避免重复加载：当父加载器已经成功加载过某个类时，子加载器无需再次加载，这保证了 JVM 中同一个类只有一个唯一的版本。
- 保证核心类库的安全性和类的唯一性：通过以上两点，双亲委派机制最终确保了 JVM 核心类库的安全性以及加载的类的唯一性。

**类加载的过程**

当一个类加载器需要加载一个类时，它会遵循以下步骤：

1. 检查是否已加载：首先，当前类加载器会检查该类是否已经被其自身或其父加载器加载过。如果已加载，则直接返回已加载的 Class 对象。
2. 委派给父加载器：如果尚未加载，当前类加载器会将加载请求委派给其父加载器。
3. 父加载器尝试加载：父加载器会在其负责的加载路径下查找并尝试加载该类。如果找到并成功加载，则返回该 Class 对象。
4. 子加载器尝试加载：如果父加载器无法加载该类（在其加载路径下未找到），子加载器才会尝试在其自身的加载路径下查找并加载。
5. 抛出异常：如果所有相关的类加载器都无法找到并成功加载该类，则最终会抛出 `ClassNotFoundException` 异常。

**获取类加载器**

- 获取当前对象的类加载器：`ClassLoader classLoader = this.getClass().getClassLoader();` 
- 获取指定类的类加载器：`ClassLoader classLoader = YourClass.class.getClassLoader();` 
- 获取系统类加载器（应用程序类加载器）：`ClassLoader systemClassLoader = ClassLoader.getSystemClassLoader();` 
- 获取扩展类加载器：`ClassLoader extensionClassLoader = ClassLoader.getSystemClassLoader().getParent();` 
- 获取引导类加载器（Bootstrap ClassLoader）：获取不到，超出了 Java 的势力范围，为 `null`

**注意**

1. 同一个类只能被同一个类加载器实例加载一次。
	- 一旦该类被成功加载到 JVM 中，该加载器会记住这个类，并且在后续需要用到这个类的时候，它会直接返回之前加载好的 Class 对象，而不会再次去查找和加载该类的字节码。

2. 类的可见性：子类加载器可以看到父类加载器加载的类，而父类加载器无法看到子类加载器加载的类。

3. 只有当两个类是由同一个类加载器加载的，并且它们的类名完全相同，JVM 才会认为它们是同一个类。
	- 类加载器不仅负责将类的字节码加载到内存中，更重要的是，它还定义了一个独立的命名空间。每一个类加载器都拥有专属的命名空间；当其加载一个类时，该类便在该命名空间中被定义。
	- 因此，当两个类拥有相同的完整类名，但却由不同的类加载器加载时，它们便处于不同的命名空间中。即使它们的字节码内容完全相同，由于加载它们的类加载器不同，JVM 也会将它们视为完全不同的类型。

4. 在 Java 中，类的加载不是在程序启动时一次性完成的，而是在需要时才进行的。
	- Java 的类加载机制采用“懒加载”策略。这意味着 JVM 并不会在程序启动时一次性加载所有类，而是在程序运行过程中，当真正需要使用某个类（例如创建实例、访问静态成员或调用静态方法）时，才会触发对该类的加载。这种机制有效地减少了 JVM 启动时的开销，并节约了内存资源。因此，在程序启动阶段，JVM 通常只会加载包含 `main` 方法的启动类以及一些核心的系统类库；其余的类则会在首次被引用时才进行加载。

### 链接（Linking）

链接发生在类被加载到内存之后，在初始化阶段之前。链接的主要目的是将已经加载到 JVM 中的二进制字节码进行进一步的处理和准备，使其能够被 JVM 正确地执行。链接阶段主要包含三个重要的步骤：验证、准备和解析。

1. 验证（Verification）：
	- 验证是链接阶段的首要步骤，其核心目标是确保加载的类的字节码文件是符合 JVM 规范的，并且不会包含任何可能导致 JVM 崩溃、数据损坏或安全漏洞的恶意指令。通过验证，JVM 能够防御恶意用户通过修改字节码来绕过安全限制。

2. 准备（Preparation）：
	- 准备阶段的主要任务是为类的静态变量（static variables，即被 `static` 关键字修饰的字段）分配内存，并设置默认初始值。
	- 需要特别注意的是，在这个阶段，仅仅是分配了内存并赋予了默认的初始值，而不是在代码中显式赋予的值。例如，一个静态整型变量 `static int count = 100;` 在准备阶段会被分配内存并初始化为 `0`，而真正将 `100` 赋值给 `count` 是在后续的初始化阶段完成的。
	- 对于被 `final` 修饰的静态变量，如果在声明的时候就进行了显式赋值（例如 `static final int VALUE = 123;`），那么在准备阶段，它们的值会被直接赋予代码中指定的值，而不会被初始化为零值。这是因为 `final` 变量一旦赋值就不能再改变。
	- 需要强调的是，准备阶段不会为实例变量（没有被 `static` 修饰的字段）分配内存和进行初始化。实例变量的内存分配和初始化发生在对象实例化的时候。

3. 解析（Resolution）：
	- 解析阶段的主要任务是将常量池中的符号引用替换为直接引用。
	- 符号引用就像是代码中使用的名字（例如类名、方法名），它们指向需要引用的其他类、字段或方法；直接引用则是这些被引用目标的实际内存地址或指针。
	- 在解析阶段，JVM 会根据符号引用中提供的名字，找到它们在内存中的真正位置。如果找不到对应的目标，JVM 会抛出错误，例如 `NoSuchFieldError` 或 `NoSuchMethodError`。这个过程通常是惰性的，即在首次使用符号引用时才进行解析。

### 初始化（Initialization）

初始化阶段是类加载的最后一个阶段，在这个阶段，JVM 才会真正开始执行类中定义的静态代码，包括对静态变量进行赋值以及执行静态代码块中的语句。

**执行类构造器 `<clinit>()` 方法**

初始化阶段的核心工作是执行由编译器自动生成的特殊方法 `<clinit>()`。这个方法并非程序员在 Java 代码中直接定义的，而是由编译器在编译时收集类中所有静态变量的赋值动作（例如 `static int value = 10;`）和所有静态语句块（`static { ... }`）中的语句，并按照它们在源代码中出现的顺序合并生成的。

值得注意的是，`<clinit>()` 方法对于类或接口来说并不是必需的。如果一个类或接口没有声明任何静态变量的赋值操作，也没有包含任何静态语句块，那么编译器可以不为它生成 `<clinit>()` 方法。

在执行一个类的 `<clinit>()` 方法之前，如果该类存在父类，JVM 会保证先执行父类的 `<clinit>()` 方法。因此，JVM 中第一个被执行 `<clinit>()` 方法的类肯定是 `java.lang.Object`。

静态代码块可以写在静态变量定义的前面，而不会出现“未定义”的错误。这是因为在链接阶段的准备阶段，JVM 已经为所有的静态变量分配了内存，并赋予了默认的初始值（通常是零值）。因此，当 `<clinit>()` 方法执行到静态代码块时，虽然静态变量的显式赋值可能还没到，但它们已经被分配了空间。

**触发初始化（主动使用）**

- 创建类的实例：通过 `new` 关键字创建类的对象时。
- 访问或修改类的静态字段：当程序访问或修改类中声明为 `static` 并且不是 `final` 的字段时。
- 调用类的静态方法：当程序调用类中声明为 `static` 的方法时。
- 对类进行反射操作：例如使用 `Class.forName()` 方法对类进行反射操作时。
- 初始化子类时发现父类还未初始化：当初始化一个类时，如果发现它的父类还没有被初始化，JVM 会先触发父类的初始化。
- 作为程序入口的主类启动时：当虚拟机启动时，用户需要指定一个要执行的主类（包含 `main()` 方法的类），JVM 会首先初始化这个主类。

除了以上列出的情况，所有其他对类的使用都被认为是“被动使用”，被动使用不会导致类的初始化。
