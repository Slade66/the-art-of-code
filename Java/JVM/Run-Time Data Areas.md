为了执行 Java 程序，Java 虚拟机（JVM）会将连续的内存空间划分为若干个具有不同功能的运行时数据区域。这些区域根据其生命周期和共享特性可以分为两类：

- 全局共享区域：堆和方法区在 JVM 启动时创建，在 JVM 关闭时销毁，其生命周期伴随着 JVM。这两个区域是所有线程共享的，用于存储所有线程都能访问的数据。

- 线程隔离区域：程序计数器、Java 虚拟机栈和本地方法栈是每个线程独有的。当一个新线程被创建时，JVM 会为该线程分配这些独立的内存区域；线程执行完毕后，这些区域会被自动回收，不同线程之间的数据互不影响。

根据 Java 虚拟机规范，JVM 的内存布局主要由以下几个部分构成：

## 程序计数器（The Program Counter Register）

程序计数器是 Java 虚拟机（JVM）运行时数据区域中一块很小的内存空间，唯一不会抛出 `OutOfMemoryError` 的区域。通过 `javac` 编译生成的 `.class` 文件中的字节码，每一条指令都有一个对应的地址。程序计数器总是指向当前线程正在执行的，或者即将要执行的字节码指令的地址，以便在线程切换后能够恢复回正确的执行状态。

Java 支持多线程并发执行。由于 CPU 会快速地在不同线程间切换，每个线程必须记录其当前执行到的指令位置，用于在线程重新获得 CPU 时间片时，能够准确地恢复到之前的执行点继续运行。因此，程序计数器是线程私有的。

如果当前线程正在执行的是一个本地方法，那么该线程的程序计数器的值为空（undefined）。

## Java 虚拟机栈（Java Virtual Machine Stacks）

Java 虚拟机栈（也称为方法栈）的主要作用是支持方法的执行。用于存储栈帧（方法执行时所需的数据），管理方法调用的顺序，追踪整个方法调用的链条。每当一个 Java 方法被调用时，一个新的栈帧就会被创建并压入到当前线程的虚拟机栈中。当方法执行完毕并返回时，对应的栈帧就会被弹出。

每个栈帧都对应一次具体的方法调用。每个方法从调用开始到执行完成的过程，都对应着一个栈帧在虚拟机栈里从入栈到出栈的过程。不同调用（即使是同一个方法的递归调用）拥有独立的栈帧，保证了局部变量等的隔离性。当前正在执行的方法对应的栈帧位于该线程 JVM 栈的顶部，称为“当前栈帧”。

栈帧的内容：
- 局部变量表（Local Variable Table）：
	- 用于存储方法参数和方法内部定义的局部变量的内存空间，它的容量以槽（Slot）为单位。
	- 一个 Slot 能存放一个 `boolean`、`byte`、`char`、`short`、`int`、`float`、`reference`（对象引用）或 `returnAddress` 类型的数据。对于 `long` 和 `double` 这种 64 位的数据类型，则需要两个连续的 Slot 来存储。
	- 对于实例方法（非 `static` 方法），其局部变量表的第 0 个槽位（Slot）固定用于存放指向当前对象实例的 `this` 引用。方法参数和内部定义的局部变量从索引 1 开始依次存储。
	- 局部变量表所需的内存空间在编译期间就已经确定了，在方法运行期间不会改变。
	- Slot 是可以复用的。当一个局部变量离开了其作用域（例如，它定义在一个代码块 `{...}` 中，并且执行流离开了该代码块），它所占用的一个或多个槽位就可以被方法中稍后声明的新局部变量所复用，这有助于节省栈帧空间。
- 操作数栈（Operand Stack）：
	- 用于在执行方法时临时存储运算所需的操作数及产生的中间结果，是字节码指令之间进行数据交换的工作台。
	- 操作数栈可以存储 JVM 定义的各种数据类型的值。对于 `boolean`, `byte`, `short`, `char` 类型的值，在压入操作数栈之前，通常会被转换为 `int` 类型进行处理。`long` 和 `double` 类型的值占用两个栈单位（Slot），其它类型占用一个。
	- 操作数栈的最大深度在编译期间就已经计算出来了，记录在方法字节码的属性中，运行时其深度不会动态变化。
- 动态链接（Dynamic Linking）：
	- 这个区域存储着当前方法执行时所需的外部符号引用，用于指向被调用的方法和引用的字段。
	- 这些符号引用具体包括：被引用类的全限定名（例如 `com/example/MyClass`）、被引用方法或字段的名称（例如 `methodA`），以及它们的描述符（例如方法描述符 `()V` 表示无参且返回 `void`）。
	- 你可以将这个区域视为一个符号表，其中的每一条记录都对应着当前方法需要使用的一个外部资源。
	- 当程序执行到需要调用一个方法或访问一个字段的指令时，JVM 会根据当前栈帧的动态链接区域中存储的符号引用，在运行时常量池中查找对应的项。如果找到了匹配的符号引用，JVM 就会解析这个符号引用，获取到被引用方法或字段的直接引用。
- 方法返回地址（Return Address）：
	- 当一个方法被执行完毕后，程序需要知道回到哪里继续执行，方法返回地址就是用来记录这个返回位置的。
	- 当一个新方法被调用时，一个新的栈帧会被创建并压入虚拟机栈。在创建这个新栈帧的同时，调用者的调用指令的下一条指令的地址会作为被调用方法的返回地址保存起来。

JVM 栈的大小直接关系到线程能支持的方法调用最大深度。可以通过启动参数（如 `-Xss1m` 设为 1MB）调整这个大小，但这需要在调用深度和内存消耗之间权衡：栈过小易引发 `StackOverflowError`，而栈过大则会增加每个线程的内存占用。

## 本地方法栈（Native Method Stacks）

本地方法栈用于支持 JVM 调用本地方法，它为每个线程维护调用本地方法所需的栈帧。

本地方法是指在 Java 代码中使用 `native` 关键字声明的方法。这些方法没有具体的实现，它们的实现是用 C、C++ 等其它编程语言编写，并编译成本地库供 JVM 调用。

在 HotSpot 虚拟机中，为了简化实现并提高运行效率，Java 虚拟机栈（用于执行 Java 方法）与本地方法栈（用于执行 `native` 方法）被合并为一个统一的线程栈。这意味着每个线程实际上只有一个物理栈结构，它同时承担 Java 方法调用和本地方法调用的栈帧管理职责。当线程执行 Java 方法时，HotSpot 会在这个统一栈中压入遵循 Java 虚拟机规范的 Java 栈帧；而当线程调用 Native 方法时，则在同一个栈中压入符合本地平台调用约定的本地方法栈帧。虽然两类栈帧的结构不同，但它们共存于同一个线程栈中。

## 堆（Heap）

在内存管理中，堆是程序在运行时动态分配内存的区域，主要作用是存放对象实例和数组，几乎所有的对象实例都在这里分配内存，被所有线程共享，也是垃圾收集器进行垃圾回收操作最频繁的区域。

从 JDK 1.7 开始，逃逸分析已被默认启用。如果一个在方法内创建的对象没有发生“逃逸”（即其引用未被返回或传递到方法外部），那么 JVM 就有可能优化地将该对象直接在栈上分配内存。

**堆的大小设置**

JVM 堆的大小可以在启动时通过指定参数来配置，其中 `-Xms` 用于设置初始堆大小，`-Xmx` 用于设置最大堆大小，`-Xmn` 用于设置新生代大小。

如果堆内存不足以分配新对象，并且垃圾回收后也无法获得足够空间，JVM 就会抛出 `OutOfMemoryError` 错误。

### 堆的内存结构

现代 JVM（尤其是 HotSpot）为了让垃圾回收更高效，会把堆内存进一步划分为新生代（Young Generation）和老年代（Old Generation）。这是因为大多数对象创建后很快就没用了，只有少数对象能活得久。根据这个特点，JVM 会对不同“年龄”的对象使用不同的回收策略。这样一来，就能用更低的成本、更短的停顿时间，清理掉最多的垃圾，从而提升程序的运行效率。

**方法区的变化**

用于存储类信息、常量池、静态变量以及方法元数据等关键信息的内存区域（方法区）在 JDK 8 前后发生了显著变化。JDK 8 之前，这些数据存储在作为堆内存一部分的永久代（Permanent Generation）中，并受到 JVM 堆大小的限制。自 JDK 8 起，元空间（Metaspace）取而代之，用于存储相同类型的数据，但它不再位于 JVM 堆中，而是直接使用本地内存。这一转变的主要目的是为了解决永久代容易出现的内存溢出问题，因为永久代的大小通常是固定的，而元空间的大小仅受限于操作系统的可用本地内存，从而使得 JVM 可以更灵活地管理这些元数据，更有效地利用系统资源，并降低了内存溢出的风险。

#### 新生代（Young Generation）

新生代主要用于存放新创建以及生命周期通常较短的对象。

由于大多数对象的生命周期都比较短，因此新生代会频繁地进行垃圾回收。

**内部结构**

新生代内部进一步划分为三个部分：
1. 伊甸园区（Eden Space）：大部分新创建的对象首先会被分配到 Eden 区。
2. 两个幸存者区（Survivor Spaces - S0 和 S1）：通常具有相同的大小。在任意时刻，其中一个区被称为“From Space”，用于存放上一次 Minor GC 后仍然存活的对象；另一个区则被称为“To Space”，处于空闲状态，准备接收下一轮 Minor GC 筛选出的幸存者对象。Survivor 区的作用类似于缓冲区，它确保只有那些真正存活较长时间的对象才有机会晋升到老年代。S0 和 S1 区会交替担任“空闲区”和“存活区”的角色。每次执行 Minor GC 后，存活的对象会从当前的 Survivor 区转移到另一个空闲的 Survivor 区。当某个 Survivor 区被填满时，存活对象会被移动到另一个 Survivor 区，直到它们的年龄达到一定阈值，才会被晋升到老年代。

**对象分配机制**

新创建的对象实例，其内存优先在伊甸园区（Eden Space）进行分配。

为了提高在多线程并发场景下的内存分配效率，避免线程之间因争抢公共内存区域而需要频繁加锁带来的性能损耗，JVM 采用了线程本地分配缓冲（TLAB）技术。这相当于在 Eden 区为每个正在运行的线程预留了一小块私有的空间。当线程需要分配新的对象时，它会首先尝试在自己的 TLAB 中进行分配。因为这块内存是线程私有的，所以在 TLAB 内的分配操作通常非常快速，且不需要进行加锁。如果线程的 TLAB 空间不足以容纳新创建的对象，或者该对象本身尺寸过大不适合在 TLAB 内分配，亦或者当前线程没有启用 TLAB，那么 JVM 会尝试在 Eden 区中除去所有 TLAB 区域后剩余的共享空间内进行分配。由于这部分内存是所有线程都可以访问的公共区域，所以在进行分配时，JVM 可能需要采取加锁等同步措施来确保操作的线程安全。

大对象直接进入老年代：对于那些体积特别庞大的对象，JVM 还存在一种特殊的分配策略，称为“大对象直接进入老年代”或“预老化”（Pretenuring）。如果一个对象所需占用的内存大小超过了 JVM 参数 `-XX:PretenureSizeThreshold` 所设定的阈值，那么 JVM 可能会选择不经过新生代（即不尝试在 Eden 区或 Survivor 区分配），而是直接分配到老年代。这种机制主要是为了避免大对象在新生代中经历 Minor GC 时被反复复制，因为复制大对象是相对耗费资源的操作。

最终，如果在 TLAB 和 Eden 的共享区域都无法找到足够的连续空间来容纳新对象（并且该对象不满足直接进入老年代的条件），这通常表明新生代的空间已经接近饱和。这种情况是触发一次新生代垃圾回收（Minor GC）最常见的时机。JVM 会暂停应用程序线程，执行 Minor GC 来清理新生代中不再被引用的对象，释放出内存空间，以便能够成功完成当前的内存分配请求，并为后续的对象创建提供可用的内存。

**新生代垃圾回收（Minor GC / Young GC）**

执行流程：
1. 标记存活对象：首先，找出 Eden 区和当前 From Survivor 区中所有仍然存活的对象。
2. 复制到 To 区：将这些存活的对象统一复制到 To Survivor 区。
3. 处理对象年龄：在复制过程中，每个被成功复制的对象的“年龄”（记录在对象头中）会加 1。JVM 会检查对象的年龄，如果达到晋升阈值，该对象在此次 GC 中会被复制到老年代，而非复制到 To Survivor 区。
4. 处理 To 区空间不足：如果在复制过程中，To Survivor 区的空间不足以容纳所有待复制的存活对象，一部分对象会被直接提前晋升到老年代。
5. 清空回收区域：完成所有存活对象的复制和晋升后，将 Eden 区和 From Survivor 区完全清空。
6. 角色互换：最后，本次 GC 中的 To Survivor 区变为下一次 GC 的 From Survivor 区，而被清空的那个 Survivor 区则成为新的 To Survivor 区。

#### 老年代（Old Generation）

老年代主要存储两类对象：一类是生命周期较长的对象，另一类是体积较大的对象。这些对象通常在新生代经历了多次 Minor GC 后才进入老年代。因此，老年代的垃圾回收（Major GC 或 Full GC）发生的频率相对较低。

**对象进入老年代的途径**

对象从新生代进入老年代主要通过以下机制：
1. 达到晋升年龄阈值：对象在新生代的 Survivor 区中，每成功经历一次 Minor GC 且存活下来，其内部的年龄计数器就会增加。当这个年龄达到由 JVM 参数 `-XX:MaxTenuringThreshold` 设定的阈值时（该参数默认值通常是 15），对象将被移动（晋升）到老年代。
2. Survivor 空间不足：如果在 Minor GC 过程中，将存活对象复制到 To Survivor 区时，发现该区域无法容纳所有对象，JVM 为了避免内存溢出，会根据策略将一部分对象（可能包括年龄未达到阈值的对象，或者所有放不下的对象）提前晋升到老年代。
3. 大对象直接分配：当尝试创建一个体积非常大的对象，其大小超过了 JVM 参数 `-XX:PretenureSizeThreshold` 指定的值时，该对象会被直接分配到老年代。这样做是为了避免大对象在新生代的 Eden 区和 Survivor 区之间进行低效的复制操作。
