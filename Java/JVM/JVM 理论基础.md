#### 为什么称它为”虚拟机“？

JVM 通过软件模拟了一个操作系统，为 Java 程序提供独立于底层操作系统和硬件的运行环境。在开发时，我们面向的是 JVM，而不是直接与底层环境交互。由于 Java 程序运行在 JVM 这个抽象层上，只要不同操作系统提供对应版本的 JVM，同一 Java 程序就能在这些平台上无需修改地运行。就像在不同物理机器上安装相同操作系统后，能够运行该系统生态中的所有应用程序一样。因此，JVM 被称为“虚拟机”。

JVM 的核心任务是将平台无关的字节码（类似于汇编语言）实时翻译为当前运行环境（特定 CPU 和操作系统）能够识别和执行的本地机器码。传统的汇编语言与特定 CPU 的指令集和操作系统紧密绑定，因此无法跨平台。为了实现跨平台，字节码抽象了这一过程，采用了独立于硬件和操作系统的虚拟指令集，就像是一种虚拟的汇编语言或机器指令。由于 JVM 执行“虚拟”指令的特性，因此它被称为“虚拟机”。

#### 为什么要编译为字节码？

**为了跨平台**

Java 源代码（`.java` 文件）经过 Java 编译器（`javac`）编译后，不会直接生成特定平台的本地机器码，而是转化为一种与平台无关的中间形式：Java 字节码（`.class` 文件）。

字节码是一种标准化的指令集，无法直接在硬件上运行，必须由 JVM 负责将其转换为当前平台的本地指令并执行。

JVM 作为通用字节码与各类操作系统和硬件之间的桥梁，屏蔽了底层平台的差异，使 Java 程序无需根据不同平台进行修改，只要目标平台安装了相应实现的 JVM，代码就可以直接运行。

这正是 Java 实现跨平台特性的关键所在。

**语言无关性**

JVM 并不直接执行 Java 源代码，而是运行符合其规范的字节码。正是由于这种设计，JVM 的能力不再局限于 Java 本身，而是为多种语言在同一平台上运行创造了可能。

只要某种语言能将代码编译为符合 JVM 规范的字节码——无论是 Java、Kotlin、Scala、Groovy 还是 Clojure——它就能在 JVM 上运行。这些语言通过各自的编译器将源代码“翻译”成统一格式的字节码，而 JVM 只负责执行字节码本身，至于字节码的来源，它并不关心。

简而言之，字节码是 JVM 与编程语言之间的“中间层”。它屏蔽了语言差异，使 JVM 无需为每种语言做专门适配。这也是为什么我们不能将 Java 源代码直接交给 JVM 执行，而必须先编译成字节码——这一中间步骤的意义，正是为了实现语言无关性。
