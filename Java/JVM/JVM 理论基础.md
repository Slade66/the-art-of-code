#### 为什么称它为”虚拟机“？

JVM 通过软件构建了一个屏蔽了底层差异的标准化虚拟执行平台，为 Java 程序提供独立于底层操作系统和硬件的运行环境。在开发时，Java 开发者面向 JVM 编写代码，而不是直接与底层环境交互。正是因为 Java 程序（字节码）运行在 JVM 这个精心设计的抽象层之上，只要不同的操作系统（如 Windows，macOS，Linux）和硬件架构（如 x86，ARM）提供了相应版本的 JVM 实现，同一份编译好的 Java 字节码就能在这些多样化的平台上无需修改地运行。JVM 为 Java 程序处理了所有底层细节，使得它就像运行在一台专属的虚拟电脑上，因此得名“虚拟机”。

JVM 定义了自己的指令集，即 Java 字节码，就像虚拟的 CPU 指令集。Java 代码被编译成这种平台无关的字节码，而不是特定硬件（如 x86 或 ARM）的机器码。JVM 内部包含一个执行引擎，它的核心任务是将平台无关的字节码（类似于汇编语言）实时翻译为当前运行环境（特定 CPU 和操作系统）能够识别和执行的本地机器码。传统的汇编语言与特定 CPU 的指令集和操作系统紧密绑定，因此无法跨平台。为了实现跨平台，字节码抽象了这一过程，采用了独立于硬件和操作系统的虚拟指令集，就像是一种虚拟的汇编语言。由于 JVM 执行“虚拟”指令的特性，因此它被称为“虚拟机”。

#### 为什么要编译为字节码？

**执行字节码的效率远超直接解释源代码**

在解释执行过程中，直接运行源代码的效率很低，因为每次都要进行词法分析、语法分析和语义分析。就像每次看一份用“人类语言”写的菜谱，都得从头读一遍、分析结构、检查每一步是否合理，最后才能动手操作。这种重复且复杂的处理方式严重影响了执行效率。

而字节码则不同。它是一种更接近机器指令的中间格式，已经由编译器完成预处理，将源代码转化为结构清晰、标准化的指令序列。执行引擎只需按照指令逐条执行，无需再次解析，大大减少了运行时的开销。就像有人提前把冗长的菜谱整理成简洁明了的操作步骤，执行起来自然更快更高效。

相比直接解释源代码（每次执行前都需进行词法、语法分析），执行预编译的字节码避免了重复的分析开销。JVM 加载并验证字节码后，能更快地进入实际的代码执行阶段。更重要的是，字节码为 JIT 编译等运行时优化提供了基础，能够实现极高的运行时效率。因此，字节码机制显著提升了 Java 程序的运行时性能，远优于直接解释执行源代码的方式。

**为了跨平台**

Java 源代码（`.java` 文件）经过 Java 编译器（`javac`）编译后，不会直接生成特定平台的本地机器码，而是转化为一种与平台无关的中间形式：Java 字节码（`.class` 文件）。

字节码是一种标准化的指令集，无法直接在硬件上运行，必须由 JVM 负责将其转换为当前平台的本地指令并执行。

JVM 作为通用字节码与各类操作系统和硬件之间的桥梁，屏蔽了底层平台的差异，使 Java 程序无需根据不同平台进行修改，只要目标平台安装了相应实现的 JVM，代码就可以直接运行。

这正是 Java 实现跨平台特性的关键所在。

**语言无关性**

JVM 并不直接执行 Java 源代码，而是运行符合其规范的字节码。正是由于这种设计，JVM 的能力不再局限于 Java 本身，而是为多种语言在同一平台上运行创造了可能。

只要某种语言能将代码编译为符合 JVM 规范的字节码——无论是 Java、Kotlin、Scala、Groovy 还是 Clojure——它就能在 JVM 上运行。这些语言通过各自的编译器将源代码“翻译”成统一格式的字节码，而 JVM 只负责执行字节码本身，至于字节码的来源，它并不关心。

简而言之，字节码是 JVM 与编程语言之间的“中间层”。它屏蔽了语言差异，使 JVM 无需为每种语言做专门适配。这也是为什么我们不能将 Java 源代码直接交给 JVM 执行，而必须先编译成字节码——这一中间步骤的意义，正是为了实现语言无关性。
