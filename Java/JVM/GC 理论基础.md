#### GC 有什么用？

垃圾回收（Garbage Collection, GC）是 Java 虚拟机（JVM）提供的一种自动内存管理机制，旨在简化内存管理的复杂性，提高程序的健壮性。与 C 或 C++ 等需要开发者手动分配和释放内存的语言不同，Java 开发者无需关心内存的生命周期管理，可以将更多精力集中在业务逻辑上，从而有效降低内存泄漏和野指针等问题的发生风险。JVM 的 GC 系统会自动追踪堆上分配的对象，并在对象不再被使用时回收其占用的内存，避免了手动管理内存带来的复杂性和风险。

**什么是垃圾？**

在程序运行过程中，凡是不再被任何存活引用指向的对象及其占用的内存，都被视为垃圾。

**什么是野指针？**

当指针变量指向了已被释放的内存地址，但该指针未被及时置为 `NULL`，仍尝试通过它访问该内存。这种行为可能导致程序崩溃，或误修改其它代码的数据，造成难以调试的错误。

**什么是内存泄漏？**

程序中不再使用的内存未被系统回收，且原本指向该内存的指针变量要么被重新赋值指向其它位置，要么超出作用域而消失，导致这块内存无法被访问或释放，从而与程序“失联”。内存泄漏在短时间运行的程序中影响较小，但对于需要长时间运行的程序，内存泄漏会导致内存占用持续增加，进而引发系统变慢，最终因资源耗尽而无法分配新内存，导致程序崩溃。

**GC 哪儿？**

堆是几乎所有对象实例和数组的内存分配区域，也是 GC 的主要工作区域。而虚拟机栈的内存随着线程的创建和销毁自动分配和释放，不由 GC 管理。GC 在方法区主要负责回收废弃的常量和不再使用的类（类卸载）。

#### 为什么需要分代？

大多数对象在创建后很快就不再使用了，而存活时间较长的对象则更有可能继续存活下去。如果每次 GC 都扫描整个堆来查找无用对象，将浪费大量时间在那些长期存活且很少被回收的对象上，效率低下。分代收集的核心思想是将堆内存按对象的存活时间划分为不同的区域（代），并针对不同代采用不同的回收频率和算法，从而提升垃圾回收的效率。

**分代回收的工作流程**

1. 新创建的对象首先被分配在新生代的 Eden 区。
2. 当 Eden 区满，触发 Minor GC（Young GC）：
	1. JVM 会暂停应用程序线程（Stop-the-World）
	2. 扫描 Eden 区和 From Survivor 区。
	3. 找出存活的对象。
	4. 使用复制算法，将存活的对象复制到另一个空的 To Survivor 区。在复制过程中，对象的“年龄”（经历的 Minor GC 次数）会增加。
	5. 清空 Eden 区和 From Survivor 区。
	6. 交换 From 和 To Survivor 区的角色，原来的 To 区现在变成了 From 区，等待下一次 Minor GC。
3. 对象晋升：
	1. 如果在 Minor GC 中，一个对象的年龄达到了阈值（比如 15 次，可以通过 `-XX:MaxTenuringThreshold` 参数设置），它就会被移动到老年代。
	2. 如果在 Minor GC 时，To Survivor 区空间不足以容纳所有存活的对象，一部分对象（即使年龄没到）也可能直接晋升到老年代。
4. 当老年代满或方法区满，触发 Major GC（Full GC）：
	1. JVM 会暂停应用程序线程（通常比 Minor GC 长）。
	2. 通常会涉及整个堆（新生代 + 老年代），有时还包括方法区（元空间或永久代）。
	3. 可能使用不同的算法或它们的组合来进行清理（标记-清除、标记-整理）

Minor GC 发生频繁，速度快，STW 时间短，对应用程序影响较小；而 Major GC 发生较少，速度慢，STW 时间长，是性能优化的重点关注对象。

#### 核心的 GC 算法

**复制（Copying）**

- 过程：
	- 空间划分与分配：将可用内存平均划分为 From 空间和 To 空间。新对象分配在 From 空间，To 空间保持空闲。
	- 标记与复制：标记 From 空间中的存活对象，将其复制到 To 空间并紧凑排列，同时更新相关引用指向新地址。
	- 清空旧空间：复制完成后，清空原 From 空间，释放内存。
	- 角色互换：To 空间变为新的 From 空间，原 From 空间变为 To 空间，等待下一次 GC。
- 优点：实现简单、效率高（大多数对象是短命的，只需复制少量存活对象）、不会产生内存碎片。
- 缺点：内存利用率低（只有一半空间可用）；若存活对象多，复制和引用更新的成本将显著增加。

**标记-清除（Mark-Sweep）**

- 过程：
	1. 标记存活对象：从 GC Roots（如线程栈、静态变量、JNI 引用等）出发，遍历并标记所有可达的对象为“存活”。
	2. 清除无用对象：遍历整个老年代堆内存，回收所有未被标记的对象，并将其内存空间记录为可用（如放入空闲列表）。
- 优点：实现简单，不涉及对象移动。
- 缺点：会产生内存碎片。由于回收后的内存空间是不连续的，如果后续需要分配一个较大的对象，即使总的空闲内存足够，也会因为找不到足够大的连续空间而导致分配失败，进而提前触发下一次 GC。

**标记-整理（Mark-Compact）**

- 过程：
	1. 与“标记-清除”算法类似，首先标记所有存活对象。
	2. 将所有存活的对象向堆内存的一端移动。
	3. 清理掉边界以外的所有内存区域。
- 优点：解决了内存碎片问题。回收后会留下一整块连续的空闲内存，有利于后续大对象的分配。
- 缺点：移动对象的成本较高，暂停时间通常比“标记-清除”要长。
