### GC 有什么用？

垃圾回收（Garbage Collection, GC）是 Java 虚拟机（JVM）提供的一种自动内存管理机制，旨在简化内存管理的复杂性，提高程序的健壮性。与 C 或 C++ 等需要开发者手动分配和释放内存的语言不同，Java 开发者无需关心内存的生命周期管理，可以将更多精力集中在业务逻辑上，从而有效降低内存泄漏和野指针等问题的发生风险。JVM 的 GC 系统会自动追踪堆上分配的对象，并在对象不再被使用时回收其占用的内存，避免了手动管理内存带来的复杂性和风险。

**什么是垃圾？**

在程序运行过程中，凡是不再被任何存活引用指向的对象及其占用的内存，都被视为垃圾。

**什么是野指针？**

当指针变量指向了已被释放的内存地址，但该指针未被及时置为 `NULL`，仍尝试通过它访问该内存。这种行为可能导致程序崩溃，或误修改其它代码的数据，造成难以调试的错误。

**什么是内存泄漏？**

程序中不再使用的内存未被系统回收，且原本指向该内存的指针变量要么被重新赋值指向其它位置，要么超出作用域而消失，导致这块内存无法被访问或释放，从而与程序“失联”。内存泄漏在短时间运行的程序中影响较小，但对于需要长时间运行的程序，内存泄漏会导致内存占用持续增加，进而引发系统变慢，最终因资源耗尽而无法分配新内存，导致程序崩溃。

**GC 哪儿？**

堆是几乎所有对象实例和数组的内存分配区域，也是 GC 的主要工作区域。而虚拟机栈的内存随着线程的创建和销毁自动分配和释放，不由 GC 管理。GC 在方法区主要负责回收废弃的常量和不再使用的类（类卸载）。

**GC 的弊端**

1. 运行时开销：GC 需要消耗 CPU 资源来追踪对象引用、执行回收算法。
2. 应用程序暂停：GC 算法在执行时需要暂停所有用户线程，这会导致应用程序短暂停止响应。
3. 无法处理逻辑内存泄漏：GC 无法解决逻辑上的内存泄漏，即程序代码错误地持有了不再需要的对象的引用，导致这些对象无法被回收。
4. 不可预测的 STW 停顿：GC 的触发时机由 JVM 内部策略决定，开发者无法精确控制 GC 的执行时机，可能在不期望的时刻发生用户线程暂停。相比之下，C/C++ 没有 GC 暂停，更适合对延迟极度敏感、实时性要求极高的场景。

### 为什么需要分代？

大多数对象在创建后很快就不再使用了，而存活时间较长的对象则更有可能继续存活下去。如果每次 GC 都扫描整个堆来查找无用对象，将浪费大量时间在那些长期存活且很少被回收的对象上，效率低下。分代收集的核心思想是将堆内存按对象的存活时间划分为不同的区域（代），并针对不同代采用不同的回收频率和算法，从而提升垃圾回收的效率。

**分代回收的工作流程**

1. 新创建的对象首先被分配在新生代的 Eden 区。
2. 当 Eden 区满，触发 Minor GC（Young GC）：
	1. JVM 会暂停应用程序线程（Stop-the-World）
	2. 扫描 Eden 区和 From Survivor 区。
	3. 找出存活的对象。
	4. 使用复制算法，将存活的对象复制到另一个空的 To Survivor 区。在复制过程中，对象的“年龄”（经历的 Minor GC 次数）会增加。
	5. 清空 Eden 区和 From Survivor 区。
	6. 交换 From 和 To Survivor 区的角色，原来的 To 区现在变成了 From 区，等待下一次 Minor GC。
3. 对象晋升：
	1. 如果在 Minor GC 中，一个对象的年龄达到了阈值（比如 15 次，可以通过 `-XX:MaxTenuringThreshold` 参数设置），它就会被移动到老年代。
	2. 如果在 Minor GC 时，To Survivor 区空间不足以容纳所有存活的对象，一部分对象（即使年龄没到）也可能直接晋升到老年代。
4. 当老年代满或方法区满，触发 Major GC（Full GC）：
	1. JVM 会暂停应用程序线程（通常比 Minor GC 长）。
	2. 通常会涉及整个堆（新生代 + 老年代），有时还包括方法区（元空间或永久代）。
	3. 可能使用不同的算法或它们的组合来进行清理（标记-清除、标记-整理）

Minor GC 发生频繁，速度快，STW 时间短，对应用程序影响较小；而 Major GC 发生较少，速度慢，STW 时间长，是性能优化的重点关注对象。

### 核心的 GC 算法

**复制（Copying）**

- 过程：
	- 空间划分与分配：将可用内存平均划分为 From 空间和 To 空间。新对象分配在 From 空间，To 空间保持空闲。
	- 标记与复制：标记 From 空间中的存活对象，将其复制到 To 空间并紧凑排列，同时更新相关引用指向新地址。
	- 清空旧空间：复制完成后，清空原 From 空间，释放内存。
	- 角色互换：To 空间变为新的 From 空间，原 From 空间变为 To 空间，等待下一次 GC。
- 优点：实现简单、效率高（大多数对象是短命的，只需复制少量存活对象）、不会产生内存碎片。
- 缺点：内存利用率低（只有一半空间可用）；若存活对象多，复制和引用更新的成本将显著增加。

**标记-清除（Mark-Sweep）**

- 过程：
	1. 标记存活对象：从 GC Roots（如线程栈、静态变量、JNI 引用等）出发，遍历并标记所有可达的对象为“存活”。
	2. 清除无用对象：遍历整个老年代堆内存，回收所有未被标记的对象，并将其内存空间记录为可用（如放入空闲列表）。
- 优点：实现简单，不涉及对象移动。
- 缺点：会产生内存碎片。由于回收后的内存空间是不连续的，如果后续需要分配一个较大的对象，即使总的空闲内存足够，也会因为找不到足够大的连续空间而导致分配失败，进而提前触发下一次 GC。

**标记-整理（Mark-Compact）**

- 过程：
	1. 与“标记-清除”算法类似，首先标记所有存活对象。
	2. 将所有存活的对象向堆内存的一端移动。
	3. 清理掉边界以外的所有内存区域。
- 优点：解决了内存碎片问题。回收后会留下一整块连续的空闲内存，有利于后续大对象的分配。
- 缺点：移动对象的成本较高，暂停时间通常比“标记-清除”要长。

### 如何判断对象已死？

**可达性分析算法**

JVM 主要通过可达性分析算法来判断对象是否可回收，从而避免了引用计数算法因循环引用而导致的内存泄漏问题。

从 GC Roots 出发，遍历所有直接或间接被引用的对象，并将它们标记为”可达“，即存活对象。无法通过 GC Roots 访问到的对象则被标记为“不可达”，即垃圾对象，可被回收。这个过程通常通过图遍历算法实现。

**GC Roots**

它们是一些程序运行中肯定不能被回收的对象，是程序逻辑正常运行的基础：
1. 虚拟机栈（栈帧中的本地变量表）中引用的对象：比如当前正在执行的方法里的局部变量所引用的对象。
2. 方法区中类静态属性引用的对象：比如类的 `static` 变量引用的对象。
3. 本地方法栈中被 `native` 方法引用的对象：如果你的 Java 代码调用了本地 C/C++ 代码，这些本地代码可能会持有 Java 对象的引用。
4. JVM 内部的对象：如基本数据类型对应的 `Class` 对象。
5. 被同步锁（`synchronized` 关键字）持有的对象。
6. 所有当前活跃的线程对象：正在运行或准备运行的线程自身也是 GC Roots。

**Java 中的四种引用类型**

- 强引用（Strong Reference）：只要一个对象有任何强引用指向它，垃圾回收器就绝对不会回收该对象，即使系统内存空间耗尽导致抛出 OutOfMemoryError。
- 软引用（Soft Reference）：当内存充足时，垃圾回收器通常不会回收仅被软引用指向的对象。只有当系统内存即将不足，垃圾回收器才会尝试回收这些对象。
- 弱引用（Weak Reference）：在垃圾回收器执行扫描时，一旦发现某个对象仅仅被弱引用指向（即没有任何强引用或软引用指向它），无论当前内存空间是否充足，该对象都将被回收。
- 虚引用（Phantom Reference）：当一个对象仅被虚引用指向，并且垃圾回收器准备回收该对象时，会在回收动作发生之前，将这个虚引用对象自身放入与之关联的 `ReferenceQueue` 中，主要用于跟踪对象被垃圾回收的状态，从而在对象内存被最终清理前执行一些必要的后置清理操作。

### 主要的垃圾回收器

JVM 提供了多种垃圾回收器实现，它们采用不同的算法和策略，以满足不同应用场景下的性能需求。

**GC 的性能指标**

- 暂停时间：所有 GC 收集器在执行过程中都会触发 STW（Stop-The-World），即暂停用户线程。若暂停时间过长，可能导致应用卡顿、无响应，甚至引发超时等问题。因此，停顿时间是衡量 GC 性能的关键指标之一，通常越短越好。
- GC 吞吐量：指在应用总运行时间中，应用程序运行时间占总运行时间的百分比。吞吐量越高，说明 GC 占用的时间越少，CPU 资源更多用于执行业务逻辑。例如，98% 的吞吐量表示 98% 的时间用于运行应用代码，只有 2% 用于垃圾回收。

#### Serial GC

Serial GC（串行收集器）是一种单线程的垃圾收集器，无论是进行新生代（Minor GC）还是老年代（Major GC）的垃圾回收，它都只使用一个 CPU 线程。在进行垃圾回收时，它会暂停所有正在运行的应用程序线程，直到垃圾回收完成。它在新生代使用复制算法，在老年代使用”标记-整理“算法。

JVM 启动参数：`-XX:+UseSerialGC`。

它的实现简单，内存消耗较少，适用于特定场景：
- 单核 CPU。
- 内存需求非常小的应用（例如堆内存小于 100MB）。
- 对垃圾回收暂停时间不敏感的应用。
- 资源极其有限的环境。

缺点：
- 无法利用多核优势：在多核 CPU 普遍存在的今天，它无法并行执行垃圾回收，限制了回收效率。
- 较长的暂停时间：由于是单线程执行且需要暂停所有应用线程，当处理大内存堆或者应用程序产生垃圾速度很快时，暂停时间可能会变得很长，严重影响应用程序的响应能力。

在现代多核服务器上，Serial GC 几乎不被选用，但在资源极其有限的场景下，它可能是唯一的选择。

#### Parallel GC

Parallel GC（吞吐量收集器）的核心目标是最大化应用程序的吞吐量（CPU 用于运行用户代码的时间占总运行时间的比例）。它通过并行处理垃圾回收任务来减少 GC 时间，从而让应用程序代码获得更多的执行时间。

JVM 启动参数：`-XX:+UseParallelGC`。

它通过多个 GC 线程并行执行垃圾回收任务。在新生代（Minor GC）中，使用 Parallel Scavenge 算法（一种多线程的复制算法）；在老年代（Major GC）中，使用 Parallel Old 算法（一种多线程的标记-整理算法）。尽管采用了多线程，所有回收阶段仍然是完全 STW 的，这意味着在 GC 执行期间，所有应用程序线程都会被暂停。

优点：利用多核CPU并行处理垃圾回收任务，可以显著提升GC效率，缩短GC总暂停时间占用的比例，最终实现更高的应用程序吞吐量。

缺点：
- 单次暂停时间较长，影响响应性：尽管采用并行处理，Parallel GC 仍会触发 STW。特别是在执行 Full GC 或处理大内存堆时，单次暂停可能持续几秒钟甚至更长，从而直接影响应用程序的响应速度。因此，它不适用于对延迟要求严格的应用场景，如实时交易系统。
- 在单核 CPU 上，由于线程管理和同步带来的额外开销，其性能可能还不如 Serial GC。

适用场景：Parallel GC 最适合运行在多核服务器上，用于那些优先考虑高吞吐量并且可以容忍较长（如秒级或以上）STW 暂停的应用。典型的例子包括对间歇性停顿不敏感的后台任务、数据处理、批处理作业，以及计算密集型的科学计算任务。

在 Java 5/6 到 Java 8 版本期间，它是服务器端 JVM 的默认垃圾收集器。

#### Concurrent Mark Sweep（CMS）GC

CMS 的设计目标是尽量减少应用程序在垃圾回收期间的暂停时间，也就是在不打断程序执行的前提下，实现边运行边清理垃圾。

JVM 启动参数：`-XX:+UseConcMarkSweepGC`。

优点：大部分垃圾回收工作（如标记和清除对象）可与用户线程并发执行，大幅缩短了需要暂停应用程序（STW）的时间。

缺点：
- 会产生内存碎片：清完垃圾后不会整理空间，导致内存像拼图一样不连续，时间久了容易触发一次耗时很久的 Full GC。
- 可能漏掉一些垃圾：在并发标记阶段，某些对象仍被使用，因此被标记为存活，但在标记结束后实际上已经用不到了，只能等下一轮 GC 才能回收。

CMS 特别适合对响应速度要求高、无法容忍卡顿的应用。在 G1 出现之前，它几乎是延迟敏感场景的首选。如今，已被 G1、ZGC、Shenandoah 等更现代的低延迟收集器取代，在 JDK 9 中被标记为弃用，从 JDK 14 起正式被移除。
