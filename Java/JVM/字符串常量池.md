字符串常量池的作用是复用字符串字面量。它是 JVM 在内存中开辟的一块特殊区域，旨在节省内存和提升性能。当程序中需要一个内容相同的字符串时，JVM 会优先检查常量池，如果池中已有该字符串，就直接返回已有对象的引用，而不会重新创建新的对象。这样可以避免生成大量内容相同但地址不同的 `String` 对象，提升效率并降低内存消耗。

这个机制依赖于字符串的不可变特性。由于字符串一旦创建，其内容就不能更改，因此多个引用指向同一个字符串对象是安全的。哪怕一个引用“修改”了字符串，实际上也是创建了新的对象，不会影响原有对象。这正是常量池机制成立的基础。

常量池主要存储两类字符串：
1. 字符串字面量：在代码中直接写出的字符串，如 `"hello"`。
2. 通过 `String.intern()` 方法主动加入池中的字符串。

当你使用字符串字面量创建字符串（例如 `String s = "abc";`）时，JVM 会：
- 先检查常量池中是否已存在内容为 `"abc"` 的字符串；
- 如果存在，返回已有的引用；
- 如果不存在，就在池中创建一个新对象，并返回其引用。

如果你使用 `new String("abc")` 这种方式创建字符串，JVM 会强制在堆内存中重新开辟空间，生成一个新的字符串对象，即使 `"abc"` 这个内容已经存在于常量池中也不复用。只有在你显式调用 `intern()` 方法时，这个字符串才会被放入常量池或者返回已有的池中对象。

另外，如果你用变量拼接字符串（比如 `String s = a + b;`，其中 `a` 和 `b` 是普通变量而不是 `final` 常量），编译器无法在编译期确定结果，所以会在运行时创建新的字符串对象。这个新对象也不会自动进入常量池，而是直接分配在堆上。

JVM 对字符串常量池的实现经历了几次演进：
1. 在 JDK 6 及之前，常量池位于方法区的永久代（PermGen）中，受固定大小限制，且 GC 效率较低，可能导致内存泄漏或 OOM。
2. 从 JDK 7 开始，常量池移到了堆内存中，允许它利用堆的垃圾回收机制，打破了永久代的限制。
3. 到了 JDK 8，永久代被移除，替代的是元空间（Metaspace），但字符串常量池依然保留在堆中。

此外，Java 8u20 引入了字符串去重（String Deduplication）功能，这是 G1 垃圾回收器的一个特性。它会自动在后台查找堆中内容相同的字符串对象，并让它们共享底层的字符数组（`char[]` 或 `byte[]`），以减少内存使用。需要注意的是，这种去重并不会改变字符串对象本身的引用地址，与常量池的 `intern()` 机制不同。

Java 对字符串进行了专门优化，避免频繁创建新对象。在项目中，如果存在大量内容相同的字符串，推荐使用字面量或 `intern()` 方法来减少内存开销，而不是直接使用 `new String(...)`。
