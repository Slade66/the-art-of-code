## 常用方法

#### `hashCode()`

`hashCode()` 方法为基于哈希表的集合（如 `HashMap`、`HashSet`、`Hashtable` 等）提供支持。

该方法是 `native` 方法，返回一个整数，由 C++ 实现。`Object` 类的默认 `hashCode()` 实现基于对象的内存地址计算哈希值。因此，在默认情况下，每个不同的对象实例（即使它们是同一类且字段值相同）都会有唯一的 `hashCode()`，因为它们在内存中的位置不同。这种基于对象“身份”而非“状态”（字段值）的哈希值，称为身份哈希码（identity hash code）。

这些集合使用哈希算法来快速存储和检索对象。当向集合中添加对象时，它们会调用该对象的 `hashCode()` 方法来生成哈希值，该哈希值通常用于计算对象在集合内部存储结构（通常是数组，也叫“桶”）中的位置。查找对象时，集合首先计算哈希码以快速定位可能的位置，然后使用 `equals()` 方法确认对象是否相同，从而解决哈希碰撞（不同对象可能有相同的哈希码）。

只要对象的 `equals()` 方法没有修改用于比较的信息，调用同一对象的 `hashCode()` 方法应始终返回相同的哈希值。如果两个对象在 `equals()` 方法中相等，则调用它们的 `hashCode()` 方法必须返回相同的哈希值。

如果重写了 `equals()` 方法，通常也需要同时重写 `hashCode()` 方法。默认的 `equals()` 方法（继承自 `Object`）是基于内存地址比较对象。如果重写了 `equals()` 使其比较对象内容（如字段值），却没有重写 `hashCode()`，则会违反上述规则。

正确的做法是重写 `hashCode()` 方法，确保它使用与 `equals()` 方法比较时相同的字段进行计算，这样才能保证相等的对象具有相同的 `hashCode()`。

Java 提供了静态方法 `System.identityHashCode(obj)`，该方法始终返回由对象默认 `hashCode()` 实现（通常基于内存地址）计算出的哈希码，无论该对象的类是否重写了 `hashCode()` 方法。当需要明确获取对象的身份哈希码时，可以使用此方法。

### 线程协作

这三个方法必须在持有锁的前提下调用（即位于 `synchronized` 块内部），否则会抛出 `IllegalMonitorStateException` 异常。

在 JVM 中，每个对象在需要加锁时会关联一个 `ObjectMonitor`（监视器锁）。其中包含两个关键的队列：
`_entry_list`：表示正在尝试进入 `synchronized` 块但尚未获得锁的线程（处于阻塞状态）。
`_wait_set`：表示已经调用了 `wait()` 方法、主动释放锁、等待被唤醒的线程。

线程在调用 `wait()` 方法后会释放持有的锁并进入等待状态；而 `notify()` 或 `notifyAll()` 只是负责唤醒等待中的线程，并不会释放当前线程持有的锁。

#### wait()

使当前线程挂起并释放锁，并加入对象的等待队列，直到被其它线程 `notify()` 或 `notifyAll()` 唤醒。

最终会调用 JVM 内部的 native 方法 `ObjectMonitor::wait()`。

**Object.wait() 和 Thread.sleep() 的区别**

所属类不同：`sleep()` 是 `Thread` 类的方法；`wait()` 是 `Object` 类的方法。

是否释放锁：`sleep()` 不会释放当前线程持有的对象锁；wait() 会立即释放当前线程持有的对象锁。

是否必须在同步块中使用：`sleep()` 可以在任何地方调用，无需处于同步代码块中；`wait()` 必须在 `synchronized` 修饰的同步方法或代码块中调用，否则会抛出 `IllegalMonitorStateException`。

唤醒方式不同：`sleep()` 到达指定时间后自动恢复执行，无需外部干预。`wait()` 必须由其他线程调用 `notify()` 或 `notifyAll()` 显式唤醒，否则会一直处于等待状态。

作用机制不同：`sleep()` 是线程自己告诉操作系统“我先不干了”，主要用于线程的定时暂停；`wait()` 是线程告诉调度器“我等别人通知”，用于线程间的协作通信。

#### notify()

唤醒一个在该对象上调用 `wait()` 而挂起的线程。

`notify()` 只是将一个等待线程从 Wait Set 移入 Entry List，使其进入“就绪状态”。但被唤醒的线程不会立刻执行，它必须等当前线程释放锁后，才能参与锁的竞争，并在成功获取锁后继续运行。

在 `notify()` 后的 `wait()` 不会被唤醒。

无法指定要唤醒的线程。

#### notifyAll()

唤醒所有在该对象上调用 `wait()` 而挂起的线程。

`notifyAll()` 会将 Wait Set 中的所有线程移动到 Entry List。它们随后会尝试重新获取锁，但仍需要等待当前线程释放锁后，才能逐个恢复执行。

#### 虚假唤醒

**什么是虚假唤醒？**

通常情况下，只有当线程接收到 `notify()` 或 `notifyAll()` 的通知时，才会从等待队列中被唤醒并从 `wait()` 方法返回。然而，在没有任何显式的 `notify()` 或 `notifyAll()` 调用时，线程也可能意外地从 `wait()` 中返回，如同“自己”醒来一般。这种在没有被显式通知的情况下从 `wait()` 返回的现象，就是所谓的虚假唤醒。

举个比喻：想象你在银行排队（线程 wait），你本来在等窗口叫号（notify）。但这时候，有个保安（JVM/OS）拍拍你肩膀说：“你可以试着看看是不是轮到你了。”你跑过去窗口一看，不是你。于是你只能重新回到队伍里（再 wait）。

**如何解决**

为了可靠地处理虚假唤醒，务必使用 `while` 循环在 `wait()` 返回后再次检查等待的条件，而不是使用 `if` 语句。`while` 循环能够确保即使线程被虚假唤醒，也会重新评估条件是否为真，如果条件仍然为假，线程将继续等待。因此，任何使用 `wait()` 的场景都必须配合 `while` 循环来检查条件，否则程序很可能在虚假唤醒后发生逻辑错误。
