### 字符串的拼接

当使用字符串字面量进行拼接时，例如 `String s = "hello" + "world";`，Java 编译器会在编译阶段优化该操作，直接将其转换为 `String s = "helloworld";`，并将结果放入字符串常量池中。

当拼接涉及变量时，例如 `String s3 = s1 + s2;`，拼接操作发生在运行时。编译器会在底层使用 `StringBuilder` 类来实现拼接。编译后的代码类似于：`String s3 = new StringBuilder().append(s1).append(s2).toString();`。因此，拼接后的字符串是在堆内存中创建的。

当字符串拼接表达式中的所有组成部分都是编译期常量（包括字符串字面量以及初始化值为编译期常量的 `final` 变量）时，Java 编译器会在编译阶段执行常量折叠优化，直接计算出最终的字符串结果。这个结果字符串会被视为一个新的字符串字面量，并存储在字符串常量池中。因此，这种情况下拼接的结果对象与直接书写的相应字符串字面量引用的是常量池中的同一个对象。反之，如果拼接表达式中包含任何非编译期常量，拼接操作将在运行时进行（通常通过 `StringBuilder` 实现），生成的结果对象位于堆内存中。

在需要进行大量字符串拼接（特别是循环内部）的场景下，建议优先使用 `StringBuilder`（适用于单线程环境）或 `StringBuffer`（适用于多线程，但有同步开销，稍慢）。直接在循环中通过 `+` 进行字符串拼接效率较低，因为编译器会在每次循环迭代都隐式创建一个新的 `StringBuilder` 对象，这带来了较大的性能开销。相比之下，如果在循环外部手动创建一个 `StringBuilder` 对象，并在循环内部反复调用其 `append()` 方法，可以有效避免重复创建对象，从而显著提升拼接操作的性能。

### 字符串的比较

在 Java 中，字符串的比较方式分为两类：
1. 使用 `==` 运算符用于判断两个字符串变量是否引用同一个内存地址，即是否指向同一个对象。
2. 使用 `equals()` 方法按字符逐一比较两个字符串的内容，区分大小写。当两个字符串的长度相同且每个字符都一致时，返回 `true`，否则返回 `false`。调用该方法时，字符串对象不能为 `null`，否则会抛出 `NullPointerException`。为了避免空指针异常，建议使用确定非空的字符串（如字面量）来调用 `equals()`。如果希望在比较时忽略大小写，可以使用 `equalsIgnoreCase()` 方法。

### `intern()`

`intern()` 是 `String` 类的实例方法，它的作用是将当前字符串放入字符串常量池（String Constant Pool），并返回池中该字符串的引用。

调用 `intern()` 方法时，JVM 会执行以下操作：
1. 检查字符串常量池中是否已有与当前字符串内容相同的对象（使用 `equals()` 判断）。
2. 如果存在，则返回池中已有对象的引用。
3. 如果不存在，则将当前字符串的引用添加到常量池，并返回该引用。

如果字符串是通过字面量创建的，如 `String s = "abc"`，那么 `"abc"` 会被自动加入常量池。

如果字符串是通过 `new String("abc")` 创建的：
- 堆中会创建一个新的 `String` 对象，内容是 `"abc"`。   
- 字面量 `"abc"` 会确保常量池中已有该字符串。

当系统中存在大量内容相同的字符串时，使用 `intern()` 方法可以将它们的引用统一指向常量池中的同一个对象，从而避免重复创建，减少内存开销。它是一种常见的字符串去重手段，尤其适合在字符串重复率高的场景中使用。调用 `intern()` 返回的一定是常量池中的引用。

### 从 `char[]` 到 `byte[]`

在 Java 9 之前，`String` 内部使用 `char[]` 数组来存储字符。无论是简单的英文字母 `'A'`，还是复杂的汉字 `'码'`，每个字符都固定占用 2 个字节。这种做法带来了一个明显的问题：当存储像 `"ABC"` 这样仅包含英文字母的字符串时，虽然这些字符属于 Latin-1 编码范围，仅需 1 个字节即可表示，但 Java 却仍然分配了 2 个字节，相当于每个字符浪费了 1 个字节的空间，效率低下。

为了优化内存使用，Java 9 引入了紧凑字符串（Compact Strings）机制，专门解决这种浪费空间的问题。其核心思想是：让只需 1 字节表示的 Latin-1 字符，真的只占用 1 个字节的存储空间。

Java 9 中对 `String` 的内部结构进行了如下优化：
- 原来的 `private final char[] value;` 被替换为 `private final byte[] value;`
- 新增了一个编码标记字段：`private final byte coder;`，用于标识当前字符串所使用的编码方式

在创建字符串对象时，JVM 会检查字符串内容是否全部可以用 Latin-1 表示：
- 如果可以，就采用 Latin-1 编码，此时 `value` 数组中的每个字节直接表示一个字符的 Latin-1 编码值
- 如果字符串中包含至少一个无法用 Latin-1 表示的字符，则采用 UTF16 编码，此时 `value` 数组中的每两个字节组成一个字符的 UTF-16 编码

这种优化显著减少了 Latin-1 字符串的内存占用，尤其在英文字符串密集的场景下效果非常明显。

**Latin-1 字符编码**

Latin-1（即 ISO-8859-1）是一种为西欧地区设计的单字节字符编码，覆盖了该地区多种主要语言，包括西班牙语、法语、意大利语、葡萄牙语、德语和荷兰语。

### 为什么 `String` 是不可变的？

一旦创建，`String` 对象的内容就无法更改。其内部用于存储字符的数组（无论是 `char[]` 还是 `byte[]`）是 `private` 和 `final` 的，并且 `String` 类也不提供任何修改这些内容的方法。所有看似修改字符串的操作（如拼接、替换）实际上都是创建了一个新的 `String` 对象，原对象保持不变。

**不可变的好处：**
- 线程安全：不可变对象天生就是线程安全的。由于不存在修改操作，多个线程可以安全地共享同一个 `String` 实例，无需任何同步机制，也不会发生“一个线程正在读取，另一个线程正在修改”的问题。这使得在并发场景中使用 `String` 简单又高效。  
- 适合作为哈希集合的键：`String` 的不可变性保证了哈希码一旦计算就不会改变，因此可以安全地作为 `HashMap`、`HashSet` 等哈希集合的键。如果字符串是可变的，一旦修改内容，其哈希码也会随之变化，容易导致哈希冲突、定位错误，甚至丢失键值。
- 支持字符串常量池：正因为 `String` 是不可变的，JVM 才能安全地将多个相同内容的字符串常量指向同一个对象，实现字符串常量池的共享优化。如果字符串可变，多个引用共享一个对象将带来严重的副作用。

### `String` 的 `hashCode()`

`String` 类重写了 `Object` 的 `hashCode()` 方法，其返回值是基于字符串内容（即字符序列）计算得出的整数，而不是对象的内存地址。这保证了只要两个字符串的内容相同（`equals()` 返回 `true`），它们的哈希码也必然相同。

由于 `String` 是不可变类，一旦创建，其内容无法更改，因此哈希码在计算后也不会变化。为了提升性能，`String` 类内部维护了一个私有字段 `private int hash`（默认值为 0）作为哈希码的缓存。首次调用 `hashCode()` 方法时，会根据字符内容计算出哈希码，并将结果缓存在该字段中。此后再次调用时，若 `hash` 不为 0，就会直接返回缓存值，从而避免重复计算。

这使得 `String` 在作为哈希表（如 `HashMap`）的键时既高效又可靠。
