### CAS 是什么？

CAS 是 Compare and Swap（比较并交换）的缩写，是实现无锁并发编程的基础	，其实就是一种算法思想，用于在多线程并发编程中实现原子更新的乐观锁机制。

它的核心思想是：当你想更新一个共享变量时，需要先检查其是否还保持你上一次读取的值，如果一致，才执行更新操作；如果不一致，就说明它已经被别的线程修改了，当前更新失败，并进入自旋（循环重试 CAS 操作）。这种机制使得我们可以在不使用锁的情况下实现线程安全的变量更新。

它其实也应用了类似版本号的思想：先从主内存中读取值，在工作内存中进行操作，写回时判断当前值是否仍与最初读取的一致。如果不一致，说明数据已被其他线程修改。为了避免误覆盖，放弃当前更新并重试。这就是比较并交换（CAS）的机制。

Java 中的原子类（如 `AtomicInteger`、`AtomicReference` 等）底层都是基于 CAS 实现的。

### 为什么要用 CAS？

其实就是在问：为什么不用锁（`synchronized`）？

当多个线程同时操作一个共享资源时，容易产生线程安全问题。传统的做法是通过加锁来保证同步，但这会阻塞其他线程，带来性能下降，还可能引发死锁。此外，线程阻塞与切换本身也会带来额外的系统开销。

而 CAS（比较并交换）是一种避免加锁的高性能并发解决方案。它不会阻塞线程，而是通过自旋重试的方式不断尝试更新数据，从而实现线程安全。正因如此，CAS 可以在保证数据一致性的同时，大幅提升多线程并发性能。

### CAS 是如何实现的？

CAS 是一种原子指令，它包含三个操作数：

1. 内存地址 (V): 要操作的变量的内存地址。
2. 期望值 (A): 你期望该内存地址上的当前值。
3. 新值 (B): 如果内存地址上的值与期望值相等，则将该内存地址上的值更新为新值。

CAS 操作会原子地执行以下步骤：

1. 比较内存地址 V 上的当前值是否等于期望值 A。
2. 如果相等，则将内存地址 V 上的值更新为新值 B。
3. 否则，不执行任何操作。

CAS 操作会返回一个布尔值，表示更新是否成功。如果更新失败，通常需要进行重试（自旋）。

CAS 实现的伪代码：

```java
boolean compareAndSwap(Object obj, long offset, int expected, int newValue) {
    int current = readMemory(obj, offset); // 从内存中读取当前值
    if (current == expected) {
        writeMemory(obj, offset, newValue); // 值一致则写入新值
        return true; // 更新成功
    } else {
        return false; // 值不一致，更新失败
    }
}
```

这里有一个关键问题：比较和写入是分两步执行的。在多线程环境下，这两步之间可能被其它线程插队，进而导致并发安全问题。因此，用 Java 这样写的代码是不安全的。为了确保线程安全，必须将“比较”和“写入”合并为一个不可分割的原子操作。

原子操作，指的是一种最小不可拆分的操作单元，一旦开始执行，就不能被中断或打断，直到操作完整地完成为止。在并发编程中，像“比较并更新”这样的操作如果能作为一个原子整体执行，意味着整个过程在多线程环境下不会被其他线程干扰，从而保证了线程安全性。

Java 本身无法将“比较”和“更新”合并为一个原子操作，这种底层的操作只能交由 C++ 实现。JVM 会调用由 C++ 编写的底层代码，通过像 `cmpxchg` 这样的原子指令，将“比较并更新”作为一个不可分割的整体执行，确保操作的原子性和线程安全。

Java 语言本身并没有直接实现 CAS，它只是提供了相关的 API，底层依赖于 CPU 提供的原子指令支持，例如在 x86 架构中常用的 `cmpxchg` 指令。在 HotSpot JVM 中，CAS 的具体实现通常是通过 C++ 代码配合内联汇编 的方式完成的，并通过 JNI 暴露给 Java 层使用。因此，CAS 的底层实现与操作系统平台、CPU 架构紧密相关，不同平台的实现方式可能会有所差异。

为什么在方法参数中要传入内存地址并直接进行内存操作？

```nasm
; 把期望值加载到 EAX（cmpxchg 要求这么做）
mov     eax, [expected]

; 把新值加载到 EBX
mov     ebx, [newval]

; lock 前缀保证在多核环境下原子性
lock cmpxchg [value], ebx  ; 如果 [value] == EAX，则用 EBX 替换
```

因为 `cmpxchg` 这条汇编指令需要根据内存地址来读取并操作对象字段的值。在 Java 中，对象创建后会被分配到内存中，并拥有一个固定的内存地址。如果我们能够计算出某个字段相对于对象起始地址的偏移量，就可以精确定位该字段在内存中的位置。通过这个地址，我们就能直接读取或修改字段的值，从而完成类似 CAS 的比较并更新操作。

在 JDK 8 及之前的版本中，原子类的底层是通过 `Unsafe` 类实现的，其核心方法 `compareAndSwapInt(obj, offset, expected, newValue)` 是一个依赖硬件支持的原子操作。`Unsafe` 调用了底层的 `native` 方法，最终会映射到 CPU 的汇编指令（如 `cmpxchg`），实现真正的原子性。从 JDK 9 开始，Java 引入了更安全、标准化的替代方案 —— `VarHandle`，用来替代 `Unsafe` 的大部分功能。

### CAS 的常见问题

#### ABA 问题

当一个值从 A 更新为 B，之后又被修改回 A 时，CAS 操作仅仅比较最终值，会错误地认为该值在整个过程中没有发生改变。这种无法感知“值虽然最终没变，但中间经历了其他状态”的情况，被称为 ABA 问题。

举个例子：
1. 线程 A 进来时读取到了值 A，随后因为某些原因暂时被阻塞。
2. 这时线程 B 把值从 A 修改为 B，又改回了 A，然后离开。
3. 当线程 A 恢复执行后，它看到值还是 A，以为数据没有被修改，于是执行 CAS 操作并成功更新。
4. 然而它并不知道，这个值在中途其实已经被其他线程修改过了。
5. 这就是典型的 ABA 问题 —— 值虽然还是 A，但已经不是原来的 A 了！

仅仅判断变量的值是否与之前相同是不够的，因为它可能在中间被其他线程修改过，然后又被改回原值。这种情况下，CAS 操作仍然会误判为值未被修改，从而导致所谓的 ABA 问题。为了解决这个问题，需要引入“版本号机制”，通过同时比较变量的值和版本号，来判断数据在期间是否真的没有被改动。Java 中常见的解决方案是使用 `AtomicStampedReference`，它在执行 CAS 操作时，除了比较变量的当前值，还会一并比较其版本号（stamp），从而有效避免 ABA 问题的发生。

#### 自旋开销

当 CAS 操作持续失败时，线程会不断地尝试重试（自旋），这会消耗大量的 CPU 资源。在高并发且竞争激烈的场景下，多个线程可能频繁地在 CAS 操作中失败，从而陷入无休止的自旋重试状态，导致 CPU 资源被过度占用。这种情况下，不仅浪费了系统资源，甚至可能引发活锁现象。因此，CAS 更适用于线程竞争不激烈、冲突概率较低的场景。

#### 只能操作一个变量

底层的 CAS 操作本质是这样：`lock cmpxchg [memory_address], new_value`，它的操作单位就是一个“内存地址”（通常 4 字节或 8 字节），它只能对这个地址做比较并交换，无法在一个原子指令中同时比较/更新多个不连续的地址。

解决方法：把多个变量封装成一个不可变对象，然后用 `AtomicReference` 一次性替换它，变相实现多个变量同时原子更新。
