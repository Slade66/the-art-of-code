### synchronized 有什么用？

当多个线程同时想要访问同一个代码块时，`synchronized` 可以确保：同一时刻只能有一个线程进入这段被加锁的代码区域，其它线程必须阻塞等待。就像银行的取款机屏蔽门，一次只能进一个人，进去之后门就上锁，其他人只能在外面排队等待前面的人出来。

`synchronized` 在英文中的原意是“同步的”。但在编程中，“同步”并不是指多个线程同时做同一件事，而是指：当多个线程访问同一个共享资源时，必须协调一致、按顺序地进行。换个说法可能更容易理解——在 Java 中，`synchronized` 可以理解为“加锁”或“互斥访问”。这些表述更贴近它的本质：当多个线程要执行同一个代码块时，必须排队，一个接一个地进入，不能乱抢。目的是为了防止多个线程同时访问共享资源，从而引发混乱或出错。

### synchronized 怎么用？

`synchronized` 可以加在代码块、实例方法或者静态方法上。加在代码块上时，你可以指定具体的锁对象；加在实例方法上，其实就是锁住 `this`；而加在静态方法上，就是锁住整个类的 `Class` 对象。

注意：
1. 避免使用字符串作为锁：因为 JVM 的字符串常量池具有缓存机制，不同地方使用相同内容的字符串常量，可能会引用到同一个对象，导致意外的线程同步冲突。
2. 构造方法不能使用 `synchronized` 修饰：但可以在构造方法内部使用 `synchronized` 代码块。
3. `synchronized` 是可重入锁，只要是同一个锁对象，线程就可以多次进入由该锁保护的代码块或方法，而不会发生死锁。

#### 修饰实例方法

`synchronized` 关键字加在实例方法，或用于 `synchronized(this)` 代码块时，表示对当前类的实例对象（`this`）加锁。

一旦某个线程正在执行这个对象的某个同步方法，其它线程就无法再执行这个对象的任何其它同步方法了。就像你在用一部手机，虽然你用的是功能 A，别人想用功能 B 也不行，因为大家操作的是同一部手机。只要一个 `synchronized` 方法在执行，其它的同步方法就会被阻塞。而非同步的方法不受影响，仍然可以正常执行。

#### 修饰静态方法

`synchronized` 关键字加在静态方法上，或用于 `synchronized(ClassName.class)` 代码块时，实际上锁的是对应类的 `Class` 对象。

静态方法上的 `synchronized` 和非静态方法上的 `synchronized` 不会互相阻塞，因为它们锁定的对象不同。静态同步方法锁的是类的 `Class` 对象，而实例同步方法锁的是具体的对象实例。由于线程操作的是不同的资源，所持有的锁也不一样，因此不会产生竞争。锁是用来保护那些可能被多个线程同时访问、从而引发冲突的资源。

#### 修饰代码块

`synchronized` 关键字加在代码块上，表示只对指定范围的代码进行加锁，括号里的对象就是锁。

常见写法如下：
- `synchronized(this)`：锁住当前实例对象
- `synchronized(ClassName.class)`：锁住当前类的 `Class` 对象（类级别锁）
- `synchronized(lock)`：锁住自定义的锁对象

### synchronized 的底层原理

#### 锁方法和锁代码块的区别

在使用 `synchronized` 修饰方法和代码块时，JVM 的处理方式是不同的：

- 修饰方法（同步方法）：不会在字节码中生成 `monitorenter` 和 `monitorexit` 指令。JVM 通过方法的访问标志（flags）中的 `ACC_SYNCHRONIZED` 标识来判断该方法是否需要同步。

- 修饰代码块（同步代码块）：在字节码中会明确生成 `monitorenter` 和 `monitorexit` 指令。进入同步块时执行 `monitorenter` 来获取锁；通常会配备两个 `monitorexit` 指令，这是为了确保无论同步块是正常执行结束还是在执行过程中抛出异常，都能正确释放锁，从而避免死锁的发生。

#### Monitor 是什么？

Java 中的“监视器（Monitor）”就是我们通常所说的“锁”，它的底层实现由 JVM 中的 C++ 类 `ObjectMonitor` 完成。每个 Java 对象在需要被同步时，都会关联一个 `ObjectMonitor` 实例，用于管理该对象的同步行为。其中一个关键字段是 `_owner`，它记录了当前持有该锁的线程。在竞争锁时，JVM 会检查这个字段来判断是否可以获取锁。

#### monitorenter 的执行过程

当执行 `monitorenter` 指令时，JVM 会尝试获取该对象的监视器锁：

1. 检查锁状态：判断锁是否被其他线程持有。
2. 如果锁未被占用：将 `_owner` 设置为当前线程，锁计数器加 1。
3. 如果锁已被当前线程持有：说明是可重入锁，继续加 1。
4. 如果锁被其他线程持有：当前线程进入阻塞或自旋，等待锁释放。

这也是为什么 Java 的 `synchronized` 是可重入锁的原因——同一线程可以多次获得同一把锁，每次进入锁时计数器加 1，退出时减 1，直到计数器为 0 才真正释放。

#### 为什么 wait/notify 必须在同步代码块/方法中调用？

Java 中的 `wait()`、`notify()` 等方法实际上是 `ObjectMonitor` 的实例方法（在 C++ 层为：`ObjectMonitor::wait`、`ObjectMonitor::notify`）。也就是说，它们操作的对象就是当前的锁对象。

为了调用这些方法，线程必须已经持有该对象的监视器锁。否则就会抛出 `java.lang.IllegalMonitorStateException` 异常。这就是为什么必须在 `synchronized` 的方法或代码块中才能使用这些方法的根本原因。

### synchronized 的性能

在早期的 Java 版本中，`synchronized` 被认为是性能较低的“重量级锁”。这是因为它底层依赖操作系统提供的互斥锁（Mutex Lock）实现线程同步，而线程的挂起和唤醒需要切换到内核态执行，涉及上下文切换，开销较大，影响程序性能。

为了解决这一问题，从 JDK 6 开始，JVM 对 `synchronized` 进行了多项优化，引入了如自旋锁、适应性自旋、锁消除、锁粗化、偏向锁和轻量级锁等机制。这些技术大大减少了线程在无竞争或轻度竞争情况下的阻塞与唤醒次数，从而提升了同步效率。

有了这些优化，`synchronized` 能根据实际的线程竞争情况动态升级锁的实现方式，从而在保证线程安全的前提下，最大限度地减少性能损耗。

需要注意的是，锁的升级是单向的，不会降级。也就是说，一旦从轻量级锁升级为重量级锁，就不会再退回。这种策略的设计目的是避免频繁的锁状态转换，提高锁操作的整体效率。

总的来说，现代 Java 中的 `synchronized` 经过多项优化后，性能已经有了显著提升，完全可以放心地应用于实际项目中。
