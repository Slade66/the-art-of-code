乐观锁和悲观锁是并发控制的一种策略，常用于数据库或多线程编程中，用于解决多个线程或事务同时访问同一资源时的数据一致性问题。

### 悲观锁（Pessimistic Lock）

悲观锁假设“别人一定会修改数据”，因此在访问数据之前就会加锁，以防止并发修改。这样一来，当其它线程试图获取该资源时，必须等待当前持有锁的线程释放锁，导致它们被阻塞。换句话说，无论是读取还是写入数据，悲观锁都会先加锁，每次只允许一个线程访问共享资源。虽然这提高了数据的安全性，但也会降低系统的并发性能。

悲观锁以牺牲性能为代价来保障数据的安全性，适用于对数据一致性要求较高的场景，尤其是在写操作频繁的情况下更为合适。

在 Java 中，像 `synchronized` 和 `ReentrantLock` 等独占锁，就是悲观锁思想的具体实现。

MySQL 中的 `SELECT ... FOR UPDATE` 就是一种悲观锁的体现，它会对查询到的数据加上排他锁，直到当前事务提交为止。

### 乐观锁（Optimistic Lock）

乐观锁认为“别人一般不会修改数据”，因此在操作数据时不会加锁，线程可以持续执行，无需等待。它的核心思想是在更新数据时进行校验，判断数据是否在此期间被其它线程修改。如果更新失败，就说明在读取数据之后，该数据已经发生了变化，原先读取的数据已经失效，此时需要重新获取数据并重试。乐观锁通常适用于读操作较多、写操作较少的场景。

乐观锁的实现方式主要有两种：版本号机制和 CAS（Compare And Swap）算法，其中 CAS 使用得更广泛一些。

#### 版本号

数据库级别的乐观锁实现相对简单，但性能表现一般。其常见做法是在数据表中增加一个版本号字段（version），每次更新时通过 SQL 语句检查当前版本号是否与读取时一致，只有一致时才执行更新；否则说明数据已被其它线程修改，需要重试更新操作直到成功。该机制能够有效避免线程使用过期数据覆盖其它线程已更新的新数据的问题。

```mysql
-- 1. 查询当前库存和版本号
SELECT stock, version FROM product WHERE id = 1;

-- 假设查询结果：
-- stock = 5, version = 3

-- 2. 执行更新
UPDATE product
SET stock = stock - 1,
    version = version + 1
WHERE id = 1 AND version = 3 AND stock > 0;
```
