在读操作非常频繁的情况下，写线程可能一直得不到执行的机会，这就是写锁饥饿问题。因为在读锁持有期间，写锁是无法获取的。为了既能保证高并发读操作，又能在必要时进行写操作，`StampedLock` 提供了乐观读锁机制。它支持无锁读取，只要读取过程中没有写入发生，就不会产生阻塞。

`StampedLock` 适用于极致的读多写少场景，其最大优势在于支持乐观读——也就是说，在读操作远多于写操作的情况下，可以不加锁直接读取数据，从而大幅提升系统性能。它还允许读操作与写操作并发进行，并通过一个 stamp（时间戳）来校验读取期间是否有写入发生；如果验证失败，则认为数据已失效，并自动回退为悲观读锁重新读取。

需要注意的是，`StampedLock` 不可重入，使用相对复杂，开发者需要手动管理 `stamp` 的获取和释放。因此，它通常只在对性能要求极高的场景中使用。

`StampedLock` 是 Java 8 引入的一种高级读写锁，支持三种模式：
- 写锁（write lock）：独占式锁，在写锁持有期间，其他线程无论是读还是写都无法获取锁，保证数据修改的原子性和一致性。
- 悲观读锁（read lock）：共享式锁，多个线程可以同时读取，但在读操作未完成前，写线程必须等待，典型的“没读完，不许写”。
- 乐观读锁（optimistic read）：通过 `tryOptimisticRead()` 获取一个数据的快照，是一种非阻塞、无加锁的读取方式，本质上是“裸读”。它不会阻塞写线程，因此读取完成后必须通过 `validate(stamp)` 方法验证读取期间是否有写操作发生。如果验证失败，则回退为传统的悲观读锁重新读取。

### 原理

`StampedLock` 内部维护了一个 `long` 类型的字段，通常称为版本号（state），用于记录锁的状态变化。每当发生写操作时，这个版本号就会递增，用于追踪数据在某个时间段内是否被修改过。

当调用 `tryOptimisticRead()` 执行乐观读时，`StampedLock` 会返回当前版本号的快照，称为 `stamp`，表示当前读取的“数据版本”。由于乐观读本质上是无锁的“裸读”，不会阻塞写线程，因此读取完成后，必须通过 `validate(stamp)` 进行校验，以确认数据在读取期间是否没被动过。

`validate(stamp)` 的原理是将当前的版本号与读取开始时获取的 `stamp` 进行比对：如果两者一致，说明读取期间没有写操作发生，数据是可靠的；如果不一致，则说明数据在读取过程中已被写线程修改，`stamp` 已失效，此时需要回退为悲观读锁，重新读取以保证数据一致性。
