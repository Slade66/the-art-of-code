### 是什么？

AQS（AbstractQueuedSynchronizer）是 Java 并发包 `java.util.concurrent.locks` 中的一个抽象类，是实现各种锁的基础。

设想一下：如果你要自己实现一个 `ReentrantLock`，会遇到很多棘手的问题，比如：
- 线程争抢资源时，怎么实现排队？
- 没抢到锁的线程，如何挂起以节省系统资源？
- 资源释放后，怎么判断应该唤醒哪个线程？

自己实现不仅复杂，还会产生大量重复代码，既低效又容易出错。

不同的锁（如 `ReentrantLock`、`Semaphore`、`CountDownLatch`）在加锁和释放方面各有差异，但它们的排队、阻塞、唤醒等底层机制是通用的。AQS 将这些通用逻辑抽取并封装为一套模板，让所有锁或同步器都能复用“线程排队 + 等待 + 唤醒”的功能，而无需重复造轮子。子类只需关注自身的加锁和释放逻辑，底层的排队、阻塞、唤醒都交给 AQS 来管理。本质上，AQS 就是通过一个共享变量 `state` 和一个 FIFO 等待队列，统一管理等待线程的基础代码。

### 原理

获取锁的流程（以独占模式为例）：
1. 线程尝试获取锁；
2. 如果获取失败：
	- 线程会被封装成一个 `Node` 节点，加入到双向 FIFO 等待队列的尾部；
	- 入队后会再次尝试获取锁；
	- 如果仍然失败，则调用 `LockSupport.park()` 挂起线程，进入阻塞状态；

释放锁的流程：
1. 持有锁的线程执行释放操作；
2. AQS 会唤醒等待队列中的下一个节点，通过调用 `LockSupport.unpark()` 使该节点中的线程从阻塞状态恢复；
3. 被唤醒的线程会重新尝试获取锁，如果获取成功则继续执行，否则将再次进入阻塞状态。

`state` 是 AQS 中用于表示共享资源状态的核心变量。其值为 0 表示资源空闲，大于 0 表示资源已被占用。为了确保在并发环境下对 `state` 的修改具有原子性，AQS 采用了 CAS 操作。之所以使用 `int` 而非 `boolean`，是为了支持更复杂的同步场景，例如：共享锁需要记录当前已有多少线程成功获取了资源。

当线程获取锁失败时，会被封装成一个 `Node` 节点，加入到等待队列的尾部。该队列是一个 FIFO 双向链表结构，每个节点包含等待线程的引用（`waiter`）、节点状态以及前驱和后继指针等信息。队列的头节点是一个“哨兵节点”（dummy node），本身不代表任何真实线程。
