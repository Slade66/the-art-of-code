## 范例
	- **一个好的 API 接口：**`/api/v1/{资源名}/{接口名}`
- ## 必须包含通用前缀 /api
	- **清晰分离：**
		- 很多应用是混合型的，既提供给浏览器访问的 **Web 页面**（例如 `/login`、`/dashboard`），也提供给程序调用的 **API 接口**。
			- `GET /dashboard` $\rightarrow$ Web Server (返回 HTML)
			- `GET /api/users` $\rightarrow$ API Gateway $\rightarrow$ Microservice (返回 JSON)
			- `GET /images/logo.png` $\rightarrow$ CDN/静态资源服务器
		- 前缀 `/api` 表示 API 接口，区分 API 接口与静态资源（如 `/images/logo.png`）、Web 页面。避免了 Web 路由和 API 路由之间的潜在冲突。
	- **便于配置网关：**
		- API 网关或反向代理负责接收外部的所有请求，并根据 URL 路径进行路由。
		- 如果所有的 API 请求都以 `/api` 开头，那么网关只需要一条简单的规则：将所有路径包含 `/api` 的请求都视为 API 流量，并进行后续的路由转发处理。非 `/api` 的请求就是静态资源和 Web 页面。
		- 如果没有这个前缀，网关的规则会变得复杂，需要枚举所有的服务路径（例如：`/users` 转发给 A 服务，`/teams` 转发给 B 服务，`/products` 转发给 C 服务...）。
- ## 必须包含版本号 /v1
	- **目的：**便于版本控制，保持接口的向后兼容性。当需要进行不兼容的修改时，创建 `/v2`，而无需影响现有客户端。
	- 当你的 API 需要进行破坏性变更时，你可以部署一个新的 `/api/v2/resource`，而让老客户端继续使用 `/api/v1/resource`，直到它们完成迁移。
- ## 必须包含所操作的资源的名称
	- **目的：**使用名词复数表示资源，它代表了由对应微服务所管理的业务领域。
	- **例子：**`POST /users`（操作的对象是用户资源集合）
	- **不要以内部的服务名称作为主体**
		- **错误示例：**`/api/user-service/users`
		- **耦合性增加：**如果你决定重命名内部服务（例如从 `user-service` 改成 `account-service`），所有外部客户端都必须修改。
		- 客户端并不关心哪个服务处理了请求，它只关心它在操作哪个业务资源。
