## Docker Swarm 是什么？
	- Docker Swarm 把多台运行 Docker 的主机组合成一个“逻辑上的单一 Docker 引擎”，可以像管理单机那样创建、部署、扩缩、更新分布式的容器化服务。
	- 比喻：把若干台机器（节点）组装成一个“虚拟集群”——你只需对这个集群发出“启动 5 个 nginx 服务”的命令，Swarm 会自动把这些容器分配到合适的节点去运行并保证它们按你要求的数量持续存在。
	- 没有 Swarm 时：你有 3 台机器都装了 Docker，要部署一个应用（比如 Web 服务 + 数据库），得手动在每台机器上跑 `docker run`，还得自己管理容器的网络、数据卷、故障重启…… 机器越多，手动操作越繁琐，还容易出错。
	- 有了 Swarm 后：这 3 台机器会被 “拉进一个集群”，你只需要向集群 “提交一个应用配置”（比如 “我要跑 2 个 Web 容器、1 个数据库容器”），Swarm 会自动把这些容器分配到合适的机器上运行，还会监控它们的状态 —— 如果某台机器宕机，Swarm 会自动在其他机器上重启容器，不用你手动干预。
	- **集群统一管理：**你可以在任意一台集群机器执行命令，管理整个集群的容器。
	- **容器高可用：**
		- 你可以指定一个容器需要 “运行的副本数”（比如 3 个 Web 容器），Swarm 会确保始终有这么多副本在运行。
		- 如果其中一个容器因为机器宕机、进程崩溃而停止，Swarm 会自动在其他健康的机器上启动一个新的副本，保证应用不中断。
	- **内置负载均衡，不用额外装 Nginx：**
		- Swarm 会为每个 “服务”（一组相同的容器，比如 Web 服务）分配一个虚拟 IP（VIP）；
		- 当你访问这个 VIP 时，Swarm 会自动把请求分发到集群中所有健康的 Web 容器上，实现负载均衡；
		- 而且容器的 IP 可能会变（比如故障重启后），但 VIP 是固定的，不用手动更新配置。
	- **跨主机通信：**
		- Swarm 提供 “Overlay 网络”，让跨机器的容器可以像在同一台机器上一样通信。
- ## Docker Swarm 的核心概念
	- **服务发现：**Swarm 带内置 DNS，服务可以通过服务名解析到对应任务的 IP。
	- **Overlay 网络与 ingress（路由网格）：**Swarm 常用 overlay 网络实现跨主机容器互联；ingress network + routing mesh 负责将外部访问路由到任意节点上的服务实例，实现集群级别暴露端口和负载转发。
	- 期望状态（desired state）
		- 你声明“我要 5 个副本”，Swarm 会持续监控并确保集群上始终有 5 个运行中的任务（若某节点挂掉，会在别的节点重建）。
	- 任务（task）
		- 服务的单个实例（即容器 + 运行在某节点上的执行单元）。
	- 服务（service）
		- 在 Swarm 上部署的应用单元。定义了镜像、要运行的副本数、端口映射、网络等。Swarm 会把服务拆成若干个任务（task），每个任务对应一个容器实例。
	- 集群（swarm）
		- 多个节点组成的整体，内部通过 Raft 协议在 manager 间同步集群状态。
	- 节点（node）
		- 参与 Swarm 的主机。分为 manager（管理节点）和 worker（工作节点）。manager 负责决策与协调（比如调度、状态存储）；worker 负责运行容器任务（tasks）。
