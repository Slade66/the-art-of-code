- 持续交付关注的是软件发布，让软件**一键发布**：发布过程自动化、标准化，软件随时准备好上线。
- 手动部署容易出错，高风险、压力大、耗时，操作繁琐。导致大家都不敢频繁更新版本，可能很久才敢更新一次，发布频率低。
- CD = 所有的部署步骤（除了最后的“批准上线”按钮）都是自动化的。
- 测试通过 -> 自动构建 Docker 镜像 -> 自动删除旧容器并启动新容器（部署上线）。
- 持续部署（Continuous Deployment）是更进一步，连最后的按钮都省了，全自动上线，但持续交付通常指保留了人工确认这一步
- **例子：**
	- **以前：**
		- 代码开发好了，得登录服务器，再手动一步一步部署。每次一个新版本都得这样手工操作。
	- **之后：**
		- **CD 流程：**
			- 小张把代码推送到 `release` 分支。
			- CI 工具跑通测试后，触发 CD 流程。
			- CD 工具自动将代码打包，并自动部署到一个与生产环境一模一样的“预发布环境”。
			- 自动化测试再次运行，验证在新环境里一切正常。
		- **上线时刻：** 运维老赵只需登录 CD 系统，看到屏幕上显示“预发布环境验证通过”。
		- **操作：** 老赵点击了一个大大的 **“部署到生产环境”** 按钮。
		- **结果：** CD 系统自动完成了所有备份、替换、重启、配置更新的工作。整个过程只用了 5 分钟，且完全没有人工失误。软件可以**随时、高频**地发布，哪怕一天发布 10 次也没问题。
