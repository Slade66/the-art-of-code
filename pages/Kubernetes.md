## Pod
	- Pod 是 Kubernetes 中最小的“可运行单元”，Pod 是 K8s 眼里最小能管理的东西，Kubernetes 管理的是 Pod 而不是单个容器。
	- **Pod 与容器的关系：**
		- ```
		  +---------------------------+
		  |          Pod              |
		  |  +---------------------+  |
		  |  |     Container A     |  |
		  |  +---------------------+  |
		  |  +---------------------+  |
		  |  |     Container B     |  |
		  |  +---------------------+  |
		  |   Shared Network, Volumes |
		  +---------------------------+
		  ```
		- Pod 是“容器的容器”，可以包含一个或多个容器。容器像“房间”，Pod 像“一个套间”。
		- Pod 内的所有容器共享网络（IP、端口）和存储卷。
		- 容器本身只负责运行应用，而 Pod 提供它们运行的环境。容器跑应用，Pod 跑容器；Pod 是容器的“家”。
		- **例子：**
			- 假设你有一个网站服务，需要处理用户请求，同时你还想收集网站日志发送到集中系统。
			- ```
			  +-----------------------------------+
			  |               Pod                 |
			  |  Shared Network & Storage Volume  |
			  |                                   |
			  |  +---------------------------+    |
			  |  |     Web App Container     |    |
			  |  |  (处理用户请求 & 写日志)   |    |
			  |  +---------------------------+    |
			  |                                   |
			  |  +---------------------------+    |
			  |  |   Log Collector Container |    |
			  |  |   (读取日志并发送到远程)   |    |
			  |  +---------------------------+    |
			  +-----------------------------------+
			  ```
			- **主容器（Web 应用）**
				- 容器 A 运行你的 Web 服务器（比如 Nginx 或 Node.js 服务）。
				- 它会产生访问日志和应用日志。
			- **侧车容器（日志收集）**
				- 容器 B 运行一个日志收集工具（比如 Fluentd）。
				- 它监控同一个 Pod 内的共享卷，把 Web 应用生成的日志发送到远程系统。
			- **它们放在同一个 Pod 的好处：**
				- **共享卷：**容器共享网络和存储卷，所以日志容器可以直接读取 Web 容器的日志文件，不需要网络传输或额外配置。这样就实现了主业务逻辑和辅助功能（sidecar）的紧密协作。
	- **为什么 Kubernetes 管理的单位是 Pod 而不是单个的容器？**
		- Pod 设计成可以包含多个容器，是为了“紧密耦合的应用组件可以一起调度和运行”。
		- **有些应用由多个容器组成：**有些应用功能上是分开的，但必须紧密协作才能完成任务。Pod 把这些容器打包在一起，保证紧密耦合的它们总是一起调度、一起销毁。
		- **共享资源：**Pod 内的容器共享网络（同一个 IP 和端口空间）和存储卷。容器之间可以直接通过 `localhost` 通信，不需要额外的服务发现或网络配置。共享卷让它们能方便地读写同一份数据。
